# v0.25.0 - 2026-07-11

## 前情与承接

`v0.25.0` 承接 `v0.20.0` 引入的 MCP 客户端与工具注册链路（自研 JSON-RPC 2.0 stdio 客户端、`mcp__{server}__{tool}` 命名空间、tool catalog 懒加载注册）。当时的状态：

1. 客户端 / MCPManager / API 路由（`/mcp/servers`、`/mcp/tools`、`/mcp/servers/reload`）已经齐备；
2. 但仓库内没有任何真实可跑的 MCP 服务器——示例配置指向不存在的 `mcp-servers/filesystem/index.js` 且 `enabled=false`，整条链路从未端到端验证过；
3. `initialize` 握手缺少 MCP 规范必填的 `clientInfo` 字段，对官方 SDK 实现的服务端会被 pydantic 校验直接拒绝，即"接上真服务器就断"。

本次把**客户端 ↔ 真实 MCP 服务器**整段链路首次打通，并修复对抗性审查发现的 4 个协议 / 健壮性缺陷。

## 版本概览

1. **首个真实 MCP 服务器落地** — 新增 `mcp-servers/utils`，基于官方 MCP Python SDK（FastMCP）实现，提供 `get_current_time` / `calculate` / `generate_uuid` 三个工具，开箱即用；
2. **客户端协议修复** — `initialize` 请求补齐 `clientInfo`；stdio 管道强制 UTF-8（不再跟随系统 locale）；
3. **健壮性修复** — 配置中 `timeout_seconds` 真正生效、握手失败回收子进程、连接失败输出 warning 日志、相对路径统一以仓库根目录为基准解析。

## 主要更新

1. 新增 `mcp-servers/utils/server.py`：官方 SDK 服务器，工具名满足 `mcp__{server}__{tool}` ≤64 字符约束；`calculate` 用 AST 白名单求值（无 eval，指数/操作数大小设限），安全性经专项攻击验证；
2. `MCPStdioClient.connect()` 的 `initialize` 参数补 `clientInfo`（`app_name` + `app_version`）——这是此前无法对接官方 SDK 服务端的硬性障碍；
3. `config/mcp_servers.example.json` 新增可直接使用的 `utils` 条目；真实配置 `config/mcp_servers.json` 加入 `.gitignore`（与 `.env` 同策略）；
4. README 新增「启用 MCP」折叠段（依赖安装 → 配置 → 开关 → 验证端点），并修正「断线重连」的过时表述。

## 问题现象

1. 官方 SDK 实现的 MCP 服务端在 `initialize` 阶段返回校验错误，连接永远失败；
2. 配置里把 `timeout_seconds` 调大对慢服务器无效——调用仍在全局 30s 处被 watchdog 杀死，且此后该服务器所有调用永久 `unavailable`，只能手动 reload；
3. 中文等非 ASCII 工具描述在遗留 locale（如 `zh_CN.GB2312`）下触发 `UnicodeDecodeError`，连接报错；
4. 对错误配置的服务器每次 reload 泄漏一个孤儿子进程；连接失败零日志，只能靠 `GET /mcp/servers` 的 `last_error` 排查；
5. 从非仓库根目录启动 uvicorn（systemd / IDE / `--app-dir`）时，相对路径的配置文件静默找不到，MCP 凭空消失。

## 根因分析

1. `initialize` 参数缺少 MCP 规范（2024-11-05）必填的 `clientInfo`，宽松服务端不校验所以此前未暴露；
2. `MCPStdioClient` 存了 `timeout_seconds` 但从未使用，所有请求硬编码全局 `mcp_init/call_timeout_seconds`；
3. `subprocess.Popen(text=True)` 未指定 `encoding`，管道编码跟随 `locale.getpreferredencoding()`，违反 MCP stdio 传输的 UTF-8 强制要求；
4. `_connect_one` 捕获握手异常后既不 `disconnect()` 回收子进程，也不打日志；失败的 client 不进 `_servers`，`shutdown_all` 无法触达；
5. 配置路径与服务器 `cwd` 以进程当前目录解析，而非项目根目录。

## 修复内容

1. `initialize` 补 `clientInfo`（`app/services/mcp_client.py`）；
2. 请求超时改为 `max(per-server timeout_seconds, 全局默认)`——只放宽不收紧，配置项恢复语义；
3. `Popen` 显式 `encoding="utf-8", errors="replace"`；
4. `_connect_one` 异常分支补 `client.disconnect()` + `logger.warning`；连接成功输出 info 日志；MCP 启用但配置缺失时输出 warning；
5. `load_config` 与服务器 `cwd` 的相对路径统一按 `PROJECT_ROOT` 解析，任意启动目录行为一致。

## API / 数据模型 / 配置变更

1. 无 API 变更；工具命名与注册链路保持兼容；
2. `.env.example` 新增 MCP 配置段（`MCP_ENABLED` / `MCP_CONFIG_PATH` / 超时项）；`APP_VERSION` 示例值同步为 0.25.0；
3. `app/core/config.py` `app_version` 0.24.5 → 0.25.0（修复版本号长期滞后于 release 的不一致）；
4. 新增 `mcp-servers/requirements.txt`（`mcp>=1.20`，仅服务器侧依赖，主应用保持零 AI 框架依赖）。

## 兼容性说明

1. 超时语义变化：显式配置了 `timeout_seconds` 且大于全局默认的服务器，慢调用不再于全局上限被杀（这是修复而非破坏）；小于全局默认时以全局为准，现有行为不变；
2. 路径语义变化：此前依赖"从非根目录启动 + 相对路径恰好指向别处"的部署（几乎不可能存在）会受影响；从仓库根目录启动的常规部署完全不变；
3. Docker 镜像暂未打包 `mcp-servers/` 与 MCP 依赖，容器内启用 MCP 需自行扩展镜像（README 已注明）。

## 验证结果

1. 全量测试通过：`tests/` 427 passed, 1 skipped（此前因缺配置跳过的用例现可运行）；
2. E2E 脚本（MCPManager 直连）：连接 → 发现 3 工具 → 三工具调用 → 错误路径（`isError` 正确回传、服务器存活）→ 关停，全部通过；从 `/tmp` 非根目录运行同样通过（路径修复实测）；
3. `LC_ALL=zh_CN.GB2312` 下连接成功、3 工具可发现（UTF-8 修复实测，修复前可复现 `UnicodeDecodeError`）；
4. 真实 uvicorn + HTTP 验证：`POST /api/v1/mcp/servers/reload` 返回 `servers_loaded:1, tools_discovered:3`；`POST /api/v1/mcp/tools/mcp__utils__calculate/test` 正确执行并回传 `structuredContent`；
5. 多智能体对抗审查：3 视角（协议互操作 / 安全 / 部署可移植）共 12 项原始发现，逐项对抗验证后确认 7 项——4 项已在本版本修复，1 项为".gitignore 正确"的正面结论，2 项转入后续建议。

## 后续建议

1. **Docker 镜像 MCP 支持**：镜像打包 `mcp-servers/` + 安装其依赖 + compose 转发 `MCP_ENABLED` + 容器专用配置（可借助已有的 `${VAR}` 展开机制）；
