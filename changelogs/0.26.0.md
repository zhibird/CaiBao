# v0.26.0 - 2026-07-11

## 前情与承接

`v0.26.0` 承接两条线：其一是 `v0.19.0` 奠定的 Agent 引擎（Function Calling 循环 + SSE 流式 + `run_mode` 分发），其二是 `v0.22.0` Phase 4 预留但从未启用的"外部代理"设想（`peer_agent_configs` 空表与两个无人读取的 flag）。当时的状态：

1. Agent App 只有两种运行模式：内置 `agent_auto`（自研 FC 循环）与 `workflow`（顺序节点）；
2. "把整个 run 委派给外部代理"只存在于死代码占位，没有任何可用实现；
3. 本机已装有成熟的编码代理 CLI（Claude Code / OpenAI Codex / Pi），它们自带工具执行、沙箱与会话管理，但与 CaiBao 完全割裂。

本次把**「外部编码代理 CLI 作为 Agent 运行后端」**整条链路首次打通：三个后端、多轮续接、SSE 流式、轨迹持久化、前端零适配。

## 版本概览

1. **三个 CLI 编码代理后端** — `run_mode ∈ {claude_code, codex, pi}` 时整个 run 委派给对应 CLI 子进程，内置 FC 循环不参与；
2. **多轮会话续接** — 每会话独立工作区目录，CLI session id 落盘续接（claude `--resume` / codex `exec resume` / pi `--session`），零数据库迁移；
3. **协议归一** — CLI 的 JSONL 事件流统一解析为 `CliAgentEvent`，映射回既有 `AgentStreamEvent` SSE 协议与 `AgentStep` 持久化，前端聊天/轨迹/回放零改动兼容。

## 主要更新

1. 新增 `app/services/cli_agent_service.py`：三 runner 共享一套子进程骨架（行缓冲 JSONL 解析、stderr 后台收集、watchdog 超时 + 进程组终止防孙进程残留、prompt 一律走 stdin 防 argv 注入），子类只实现 `build_command()` / `parse_line()` 两个纯函数——新增后端（如 Gemini CLI）照抄即可；
2. `AgentService` 新增 `_run_cli_agent` / `_run_cli_agent_stream`，在 `run()` 与 `stream_run()` 两个入口同步分发；复用 `_record_step` / `_finish_run` / `_record_agent_history`，运行回放、事件总线、记忆沉淀免费获得；
3. `AgentAppService._ALLOWED_MODES` 扩充三模式，App 工作台「运行模式」下拉框同步新增三项；
4. 事件映射：CLI `text` → SSE `llm.delta`；`tool_use`/`tool_result` → `tool.started`/`tool.result` + `AgentStep(tool_call)`（含 codex 的 `apply_patch` / `mcp_tool_call` / `web_search` 条目化）；失败统一定性为 run `failed` 并透传 CLI 错误详情（含 stderr 尾部）；
5. 前端修复：`updatePendingLabel` 的选择器指向不存在的 `.message-content .message-text`，导致流式文字与工具活动标签从不显示，修正为 `.pending-text`（历史 bug，本功能依赖流式展示故一并修复）；
6. 顺手修复 `stream_run` 失败路径的 `seq` 空值崩溃（内层生成器在首个事件前抛异常时 `None += 1` 会以 TypeError 掩盖真实错误）。

## API / 数据模型 / 配置变更

1. 无新增 API 端点、无数据库迁移：新模式复用 `AgentApp.mode`（String(32)）与 `AgentRunRequest.run_mode`（自由字符串）既有字段；CLI session id 存于工作区文件 `.caibao_cli_sessions.json`，不入库；
2. `.env.example` 新增 `CLI_AGENT_*` 配置段：总开关 `CLI_AGENT_ENABLED`（默认 **false**，遵循 Phase 4 default-off 惯例）、各 CLI 路径与模型、`CLI_AGENT_EXTRA_ENV_JSON`（向子进程注入 API key 等）、工作区根目录 / 超时 / 单步输出截断；
3. `app_version` 0.25.0 → 0.26.0（`app/core/config.py` / `.env.example` / README 徽章三处同步）。

## 安全与边界

1. 委派即信任转移：CLI 代理在**自身沙箱**内自主执行工具（codex 默认 `workspace-write`、claude 默认 `acceptEdits`、pi 禁用扩展/技能发现），CaiBao 的危险工具确认流不适用于 CLI 内部动作——`dry_run` 分别映射为 claude `plan` / codex `read-only` / pi `--no-tools`；
2. **环境隔离**：子进程不整份继承服务器环境，只透传 `_ENV_ALLOWLIST` 白名单（`PATH`/`HOME`/locale/代理/CA），杜绝 `AUTH_JWT_SECRET`/`DATABASE_URL`/provider key 被 CLI 内 shell 读出；CLI 凭据走 `~/.claude`、`~/.codex/auth.json`（HOME 可达）或 `CLI_AGENT_EXTRA_ENV_JSON` 显式注入；
3. **工作区隔离**：每会话目录 `<root>/<team>/<conversation>`，路径组件经单射清洗（对丢失信息的 id 追加摘要）防跨租户碰撞；prompt 经 stdin 传递不进 argv；
4. 单步输出入库前按 `CLI_AGENT_MAX_STEP_OUTPUT_CHARS` 截断，防止 CLI 大输出撑爆 `agent_steps`；
5. **已知限制（文件系统读隔离）**：`workdir` 只设 cwd 而非读监狱，CLI 可读取宿主机进程可达的任意文件（含应用 DB、其他租户工作区），OS 层绕过应用级 `team_id` 边界。故默认关闭、仅对可信成员开放，且勿将 DB/`.env` 与工作区同路径树部署；生产多租户强隔离需为每次 run 套 OS 沙箱（详见 `docs/step-13-cli-agent-backends.md` 第 6、7 节）。

1. 全量测试通过：`tests/` 469 passed, 1 skipped（跳过为环境依赖：未配置 PostgreSQL 的迁移用例）；其中 `tests/test_cli_agent.py` 42 项（三 runner 的命令构造 / 事件解析、假 CLI 子进程集成含崩溃 / 超时降级 / exit0-无结果 / 环境变量白名单、API 级 SSE 全链路含会话续接与未启用拒绝）；
2. 真实 CLI 端到端（uvicorn + HTTP + SSE）：**codex 全通**——真实创建文件、`apply_patch` 步骤入轨迹、同会话第二轮正确回忆上一轮产物（`exec resume` 续接实测）；claude_code / pi 链路正常、失败定性正确（本机凭据限制：claude 无头凭据、pi 未配 provider key，错误信息完整透传至 run 失败原因）；
3. 多智能体对抗审查：4 视角（子进程与生成器正确性 / 安全与租户隔离 / 引擎契约与文档一致性 / 前端）逐项对抗验证全部未提交 diff，确认 13 项真实问题并全部处理——含 1 项 critical（子进程继承全量服务器环境泄漏密钥，已改环境变量白名单）、6 项 major（超时误判成功为失败、exit 0 无结果事件被静默判成功、CLI 失败未发 `run.failed` 前端误判成功、index.html 缓存导致新模式静默降级等）、6 项 minor；1 项 major（文件系统读隔离）作为已知限制记入文档与配置警示；两个关键修复补了回归测试（超时后降级、exit 0 无结果判失败、环境变量白名单）。

## 后续建议

1. **Docker 镜像支持**：镜像未打包 Node.js 与三个 CLI，容器内启用需自行扩展镜像（与 MCP 容器说明同口径）；
2. **Pi RPC 常驻模式**：`pi --mode rpc` 支持 steer / follow_up / abort 中途干预，可在现有 runner 抽象上增强为可打断的长会话；
3. **CaiBao 工具回灌**：将 CaiBao 工具目录以 MCP server 形式暴露给 CLI 代理（三个 CLI 均支持 MCP），实现知识库/记忆卡在 CLI 侧可用。
