# v0.19.0 - 2026-05-26

## 前情与承接

`v0.19.0` 承接 `v0.18.0` 的 Docker / PostgreSQL / 企业工作台稳定化结果，以及上一轮 Dify-lite Agent App 工作台的应用配置、发布、轻量 workflow、工具插件元数据和 App Eval 能力。

本阶段的目标是完成 Agent 引擎现代化 Phase 1：把 CaiBao 从“规则关键词规划 + 同步整段返回”的 Agent，升级为“LLM Function Calling + 多轮工具循环 + SSE 流式输出 + 模型路由配置”的 Agent 主链路，为后续 MCP、事件总线、多层记忆和多智能体阶段打基础。

## 版本概览

本版本主要完成：

1. Agent 工具选择从关键词匹配升级为 OpenAI-compatible `tools` / `tool_calls` 驱动，并保留关键词 fallback。
2. 新增 queued run + SSE stream 执行链路，前端可以实时展示 LLM delta、工具状态、确认请求和最终结果。
3. 新增 planner / fast / vision 模型路由配置和脱敏持久化，其中 Agent 主执行链路已接入 planner 路由。
4. 完整补齐危险工具确认、App 工具白名单、API key 防泄漏、stream replay、fallback confirmation 等边界。
5. 更新数据库迁移、前端工作台、测试与文档，使 Phase 1 能作为可试用版本合并到 `main`。

## 主要更新

1. `LLMService` 增加非流式与流式 Function Calling 能力，支持 `tools`、`tool_choice`、`tool_calls` 解析和 SSE chunk 消费。
2. `ToolRegistry` 增加 OpenAI-compatible tool definition 映射，并使用 JSON Schema 校验工具参数。
3. `AgentService` 新增多轮工具循环：LLM 思考、选择工具、执行工具、观察结果、继续推理，直到输出最终答案或触发确认。
4. 新增 `/api/v1/agent/runs` queued run 入口和 `/api/v1/agent/runs/{run_id}/stream` SSE 入口，兼容已有 `/api/v1/agent/run` 同步入口。
5. 前端 Agent / App 调用改为消费 EventSource，实时更新 pending assistant message，并在需要确认时渲染确认按钮。
6. 新增 `LLMRouter`，支持 per-run / per-app 的 planner、fast、vision 路由配置；真实 API key 只在内存执行路径使用，持久化和响应只返回脱敏快照。
7. Agent App 增加 per-app `llm_routing` 配置，输入、输出和发布快照均做脱敏清洗。
8. 危险工具确认链路支持 Function Calling loop 与 keyword fallback 两种路径，避免确认状态不可恢复。

## 处理链路 / 流程说明

1. 同步路径：
   - `/api/v1/agent/run` 创建 running run。
   - `AgentService` 解析 RAG / memory / library 上下文。
   - `LLMRouter` 解析 planner 模型。
   - `LLMService.complete_chat()` 携带工具 schema 请求模型。
   - 模型返回 `tool_calls` 后执行工具、追加 tool result，再进入下一轮。
   - 若危险工具需要确认，则保存 pending tool call 与 loop state，返回 `requires_confirmation`。

2. 流式路径：
   - `/api/v1/agent/runs` 创建 queued run。
   - `/api/v1/agent/runs/{run_id}/stream` 执行或回放 run。
   - `LLMService.stream_chat()` 使用 `httpx.stream()` 消费 SSE。
   - 后端逐步发送 `run.started`、`llm.delta`、`tool.proposed`、`tool.started`、`tool.result`、`confirmation.required`、`run.completed` 等事件。
   - 前端 `EventSource` 消费事件并更新消息区域。

3. 确认路径：
   - Function Calling loop 保存 assistant `tool_calls` 消息和 pending `tool_call_id`，确认后追加 tool message 并恢复 LLM loop。
   - keyword fallback 没有 LLM loop state 时，确认接口会走专门 fallback confirmation 路径，执行 pending dangerous tools 并完成 run。

## 安全与兼容性说明

1. API 不接受明文 `api_key` / `base_url` 作为公开 model route 配置；公开配置只接受模型名引用。
2. `model_routes_json` 和 App response 中的路由信息只返回脱敏快照。
3. App 旧数据、导入数据或内部写入中若存在 route dict，输出侧也会再次清洗。
4. 危险工具默认进入 `requires_confirmation`，`dry_run=true` 不产生副作用。
5. 对不支持 tools / stream 的模型保留关键词 fallback。
6. planner 路由已接入 Agent 主链路；fast / vision 路由配置已保留，但专用执行步骤将在后续阶段接入。
