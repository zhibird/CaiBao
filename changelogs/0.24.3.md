# v0.24.3 - 2026-06-03

## 回顾

`v0.24.3` 修复了一个隐蔽的 LLM 协议兼容性 bug：Agent 多轮 tool loop 中，第二轮 LLM 调用时 `tool_calls[].function.arguments` 为 dict 而非 JSON 字符串，导致 DeepSeek 拒绝请求（400 `invalid type: map, expected a string`）。

## 问题现象

QQ Bot 用户发起联网搜索请求 → Agent 第一轮成功调用 `web_search` → 第二轮把工具结果 + tool_call 历史发回 LLM 总结 → **400 Bad Request**：

```
LLM request failed: Failed to deserialize the JSON body into the target type:
messages[3]: invalid type: map, expected a string
```

web_search 本身执行成功，问题出在**工具执行完后**把历史消息发回 LLM 的环节。

## 根因分析

**链路：**

```
_parse_tool_calls()           ← 把 function.arguments 从 JSON string 解析为 dict
        ↓                     （方便后续工具执行）
agent 把 tool_call 原样塞回
messages[].tool_calls          ← arguments 仍是 dict
        ↓
_build_payload → DeepSeek     ← DeepSeek 要求 arguments 必须是 string
        ↓
400: invalid type: map
```

OpenAI-compatible 协议规定 `tool_calls[].function.arguments` **必须是 JSON 字符串**。`_parse_tool_calls()` 为了方便工具执行将其解析为 dict，但 agent 在构建下一轮消息时把同一个 dict 放回了 `messages[].tool_calls`，导致 provider 拒绝。

## 修复内容

在 `_build_payload` 发送前新增 `_ensure_content_is_string` 兜底检查，做两件事：

1. **content 归一化**（保留多模态 content-parts）— 非 string 非 null 的 content 序列化为 JSON 字符串
2. **tool_calls arguments 归一化** — `tool_calls[].function.arguments` 若为 dict 则转回 JSON 字符串；若为 None 则填充 `"{}"`
